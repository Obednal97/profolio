# Community Guidelines

**Effective Date:** December 1, 2024  
**Last Updated:** December 1, 2024

## 1. Welcome to the Profolio Community

Welcome to the Profolio community! We're excited to have you join our community of developers, self-hosters, and portfolio management enthusiasts. These guidelines help ensure our community remains welcoming, productive, and supportive for everyone.

## 2. Our Mission

Profolio exists to provide a **private, self-hosted portfolio management solution** for individuals and organizations who value:
- **Data Privacy:** Your financial data stays under your control
- **Open Source:** Transparent, auditable, and community-driven development
- **Self-Hosting:** Freedom from vendor lock-in and third-party dependencies
- **Professional Features:** Enterprise-grade functionality for personal use

## 3. Community Values

### 3.1 Respect and Inclusivity

- **Be respectful:** Treat all community members with kindness and respect
- **Be inclusive:** Welcome people of all backgrounds, experience levels, and perspectives
- **Be patient:** Remember that everyone was a beginner once
- **Be constructive:** Focus on helping others learn and grow

### 3.2 Collaboration and Support

- **Help others:** Share your knowledge and experience
- **Ask questions:** No question is too basic or too advanced
- **Share solutions:** Document fixes and workarounds for common issues
- **Contribute back:** Give back to the community when you can

### 3.3 Quality and Excellence

- **Write clear documentation:** Help others understand your contributions
- **Test thoroughly:** Ensure changes work before submitting
- **Follow best practices:** Maintain code quality and security standards
- **Iterate and improve:** Embrace feedback and continuous improvement

## 4. Participation Guidelines

### 4.1 Getting Started

**New to Profolio?**
- Read the [README](https://github.com/Obednal97/profolio/blob/main/README.md) for project overview
- Check the [Installation Guide](https://github.com/Obednal97/profolio/blob/main/docs/INSTALLATION.md) for setup instructions
- Browse [Discussions](https://github.com/Obednal97/profolio/discussions) for community Q&A
- Review [Issues](https://github.com/Obednal97/profolio/issues) for known problems and feature requests

**Ready to Contribute?**
- Read our [Contributing Guide](https://github.com/Obednal97/profolio/blob/main/CONTRIBUTING.md)
- Review the [Development Setup](https://github.com/Obednal97/profolio/blob/main/docs/DEVELOPMENT.md)
- Check the [Roadmap](https://github.com/Obednal97/profolio/projects) for planned features
- Look for ["good first issue"](https://github.com/Obednal97/profolio/labels/good%20first%20issue) labels

### 4.2 How to Get Help

**For Usage Questions:**
- Check the documentation first
- Search existing discussions and issues
- Post in [GitHub Discussions](https://github.com/Obednal97/profolio/discussions) under "Q&A"
- Include relevant details: OS, version, error messages, logs

**For Bug Reports:**
- Use the [Bug Report](https://github.com/Obednal97/profolio/issues/new?template=bug_report.md) template
- Include steps to reproduce the issue
- Provide system information and logs
- Check if the issue already exists

**For Feature Requests:**
- Use the [Feature Request](https://github.com/Obednal97/profolio/issues/new?template=feature_request.md) template
- Explain the use case and benefit
- Consider implementation complexity
- Be open to alternative solutions

### 4.3 Communication Channels

**Primary Channels:**
- **GitHub Issues:** Bug reports, feature requests, project planning
- **GitHub Discussions:** Q&A, ideas, general discussion, announcements
- **GitHub Pull Requests:** Code contributions and reviews

**Community Guidelines for Each Channel:**
- **Stay on topic:** Keep discussions relevant to Profolio
- **Search first:** Check if your question/issue already exists
- **Use appropriate labels:** Help organize content for others
- **Follow templates:** Use provided issue and PR templates

## 5. Contribution Standards

### 5.1 Code Contributions

**Before You Code:**
- Discuss significant changes in an issue first
- Follow the existing code style and patterns
- Ensure your development environment is properly set up
- Write tests for new functionality

**Submitting Code:**
- Create a descriptive pull request title
- Explain what your changes do and why
- Reference related issues using keywords (e.g., "Fixes #123")
- Be responsive to review feedback

**Code Quality Standards:**
- Follow TypeScript/JavaScript best practices
- Maintain consistent formatting (use Prettier)
- Write clear, self-documenting code
- Include appropriate error handling

### 5.2 Documentation Contributions

**Types of Documentation:**
- **Code Documentation:** Comments, README updates, API docs
- **User Guides:** Installation, configuration, usage instructions
- **Developer Docs:** Setup guides, architecture documentation
- **Community Resources:** FAQs, troubleshooting guides

**Documentation Standards:**
- Write for your audience (users vs developers)
- Use clear, concise language
- Include relevant examples and screenshots
- Test instructions before submitting

### 5.3 Community Contributions

**Non-Code Ways to Help:**
- Answer questions in discussions
- Improve documentation and guides
- Report bugs and test beta features
- Share your Profolio setup and experiences
- Help with translations (when available)

- Don't request or share personal financial information

## 7. Moderation and Enforcement

### 7.1 Community Moderation

Our community is moderated by:
- **Project Maintainers:** Core team members with administrative access
- **Community Moderators:** Trusted volunteers who help maintain standards
- **Automated Systems:** GitHub's built-in moderation tools

### 7.2 Reporting Issues

**How to Report:**
- Use GitHub's report feature for specific content
- Email security@profolio.app for security concerns
- Contact maintainers directly for sensitive issues

**What to Include:**
- Links to the problematic content
- Clear explanation of the issue
- Any relevant context or evidence

### 7.3 Enforcement Actions

**Progressive Response:**
1. **Warning:** Private message explaining the issue
2. **Temporary Restriction:** Limited participation for a period
3. **Permanent Ban:** Complete removal from community spaces

**Appeals Process:**
- Contact maintainers within 30 days of action
- Provide explanation and any mitigating factors
- Decisions are reviewed by multiple maintainers

## 8. Open Source Responsibilities

### 8.1 Licensing and Legal

- All contributions are licensed under the MIT License
- Ensure you have the right to contribute your code
- Respect third-party licenses and attributions
- Don't include proprietary or copyrighted material

### 8.2 Community Ownership

- The community collectively owns the project's direction
- Major decisions are discussed openly in issues and discussions
- Maintainers serve the community's best interests
- All voices are heard and considered

### 8.3 Sustainability

**Supporting the Project:**
- Star the repository to show support
- Share Profolio with others who might benefit
- Contribute documentation, code, or community support
- Sponsor development if you have the means

## 9. Privacy and Data Protection

### 9.1 Community Privacy

- Don't share personal financial information in public discussions
- Be mindful of what you include in screenshots and logs
- Respect others' choice to remain anonymous
- Use pseudonyms or redacted examples when discussing sensitive topics

### 9.2 Data Collection

**In Community Spaces:**
- GitHub collects standard usage analytics
- We don't collect additional data beyond GitHub's platform
- Your contributions become part of the public record
- You can request removal of specific content if needed

## 10. Getting Involved

### 10.1 Ways to Participate

**For Users:**
- Share your experience and use cases
- Report bugs and suggest improvements
- Help other users with questions
- Write tutorials and guides

**For Developers:**
- Contribute code improvements and new features
- Review pull requests and provide feedback
- Help with testing and quality assurance
- Maintain and improve documentation

**For Community Leaders:**
- Organize local meetups or virtual events
- Create educational content and resources
- Mentor new contributors
- Advocate for the project in your networks

### 10.2 Recognition

**Community Contributors:**
- Regular contributors may be invited to join the maintainer team
- Significant contributions are recognized in release notes
- Community members are featured in project updates
- Special badges and recognition for consistent helpers

## 11. Updates to These Guidelines

### 11.1 Community Input

These guidelines evolve based on:
- Community feedback and suggestions
- Lessons learned from community interactions
- Changes in the project's scope and scale
- Legal and regulatory updates

### 11.2 Change Process

1. **Proposal:** Suggested changes are discussed in GitHub Discussions
2. **Review:** Community members provide feedback and suggestions
3. **Decision:** Maintainers make final decisions based on community input
4. **Implementation:** Changes are implemented and communicated
